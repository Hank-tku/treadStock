## 详细分析

### 1. 用户价值 (7/10)

**现状**：
产品瞄准散户投资者的真实痛点——选股耗时、信息分散、缺乏系统化复盘。PRD 清晰定义了用户旅程：选股 -> 跟踪 -> 决策 -> 复盘 -> 预警，每个环节都有对应功能覆盖。核心理念"5 分钟内完成每日选股扫描"是一个有力的价值承诺。

**问题**：
- 用户画像基于 0 个真实用户接触，所有假设未验证。这是最大的风险——我们可能在做一款没人要的产品。
- "5 分钟内完成"的价值承诺需要端到端验证。推荐列表全市场扫描（5000+ 股票）的首次加载 PRD 要求 < 1.5s，但架构决策中全市场数据拉取被列为技术债（"首次加载耗时较长"），存在矛盾。
- F007 复盘功能被降为 Could 优先级，但复盘恰恰是 PRD 中强调的核心差异化价值之一（"建立可量化的判断评分体系"）。KR3 也依赖评分记录积累。如果复盘被弱化，核心价值闭环就断了一环。

**挑战**：
- 0 个真实用户接触，如何确保做的是真需求而不是伪需求？
- 全市场首次数据拉取在 4G 网络下 < 1.5s 是否现实？如果不现实，"5 分钟完成扫描"的价值承诺还能否成立？
- 复盘功能降到 Could，评分系统的闭环价值如何体现？

**改进建议**：
- A) 在 MVP 上线前找 3-5 个目标用户做 15 分钟访谈，验证痛点假设（0.5 天）—— **推荐**
- B) 推荐列表首次加载分批渲染：先展示缓存 + 骨架屏，后台异步拉取全市场数据（已部分实现，需在 PRD 中明确降级体验）（0.5 天）
- C) 将 F007 复盘功能从 Could 提升为 Should，至少做到"展示评分准确率"这个最小形态（1.5 天）

**如果不改的代价**：
- 零用户验证 = 高概率做错产品，2 周开发全部浪费
- 首次加载慢 = 用户第一印象差，7 秒内流失率可能超过 40%

### 2. 差异化 (5/10)

**现状**：
PRD 将竞品缺陷定义为"信息碎片化、无自动化波段分析、无复盘机制"。设计规范显示这是一个数据密集的专业金融工具，风格克制、信息密度高，AI Slop 检测全部通过。

**问题**：
- 波段分析（MA20/MA60 + 布林带）是极其通用的技术指标，同花顺、东方财富、通达信均已内置，且功能更丰富（支持自定义参数、多周期叠加、画线工具等）。PRD 说"用户依赖 Excel 手动记录"，但实际上同花顺的"条件选股"功能可以实现类似的波段低位筛选。
- "简化评分系统"（1-10 分）看起来简单直观，但评分公式（MA + 布林带 + 量比 + 趋势加权）本质上是对现有技术指标的线性组合，并没有独特的信息增量。用户如果已经会看均线和布林带，这个评分对他们价值有限。
- 新闻聚合功能（F003）本质是 RSS 聚合，财联社 App 本身就能做到，且内容更全。
- 纯客户端架构意味着所有用户计算是独立的，没有群体智慧、没有众包数据、没有网络效应。这使得产品更像一个"个人工具"而非"平台"。

**挑战**：
- 用户为什么用股势 TrendStock 而不是同花顺的"条件选股" + 自选股管理？
- 1-10 评分提供了什么增量信息，是用户自己看 K 线图得不到的？
- 没有网络效应的产品如何建立壁垒？

**改进建议**：
- A) 强化"跨 App 整合"定位：不只做分析工具，而是做"多源信息一站式聚合器"，在首屏明确传达"不用再切 3 个 App"（调整文案和首屏设计，0 天）
- B) 评分系统增加"评分变化趋势"（连续 3 天评分上升/下降），这是同花顺没有的维度——时间维度的评分走势比单次评分更有决策价值（1 天）
- C) 长期考虑：引入"用户评分 vs 系统评分"对比功能，形成独特的复盘闭环（v2 规划，不增加当前工作量）

**如果不改的代价**：
- 产品定位模糊，用户会觉得"同花顺也能做到"，无法形成切换动机
- 评分系统被忽视，沦为装饰性功能

### 3. 可行性 (8/10)

**现状**：
架构决策清晰、务实。纯客户端架构是 2 周 MVP 的正确选择，零运维成本，技术栈选型合理（Flutter + Riverpod + Drift + Dio），全部使用成熟稳定的开源库。状态机设计完整，错误路径覆盖充分，降级方案明确。

**亮点**：
- "先展示缓存，再异步刷新"的策略完美解决了冷启动 < 2s 的要求
- 6 个数据表设计合理，字段命名清晰，缓存清理策略明确
- 错误路径地图覆盖了 6 种核心异常场景，每种都有具体恢复策略
- 技术债登记诚实且有时间节点，不是无意识的负债

**问题**：
- 东方财富 API 为非官方接口，这是最大的可行性风险。PRD 死亡条件中提到"新闻数据源 API 收费或关闭"需要重新评估，但实际上东方财富 API 变更/封禁的风险比"收费"更高——它本来就不是公开 API。
- 全市场 5000 股票一次拉取（`pz=5000`），在 4G 网络下数据量可能超过 5MB，首次加载 1.5s 的 PRD 要求在弱网环境下几乎不可能达到。
- iOS 后台任务不可靠（workmanager 限制），F006 下跌预警在 iOS 上可能形同虚设。PRD 将 F006 定为 Should，但如果 iOS 用户占比较高（通常 30-40%），这个功能的价值会大打折扣。

**挑战**：
- 东方财富 API 封禁后的切换成本是多少？如果第一天就封了怎么办？
- 全市场数据首次拉取在弱网环境下的真实耗时是多少？

**改进建议**：
- A) 立即实现新浪财经作为备用数据源（架构已提到，但建议在 MVP 中就集成切换逻辑，而非等 v1.1）（1.5 天）—— **推荐**
- B) 首次安装预装一份全市场股票基础数据（代码+名称+行业），减少首次网络请求量（0.5 天）
- C) PRD 中将推荐列表首次加载的 P95 指标从 < 1.5s 调整为 < 3s（缓存命中保持 < 1.5s），更贴近现实（0 天，仅文档修改）

**如果不改的代价**：
- 东方财富 API 封禁 = 产品完全不可用，MVP 失败
- 弱网加载慢 = 首次用户体验差，可能直接卸载

### 4. 增长潜力 (6/10)

**现状**：
PRD 定义了明确的成功判断标准：2 周后日活 60%+ 查看推荐/关注，关注列表平均留存 > 5 只。死亡条件也很清晰（DAU < 20 则重新评估）。

**问题**：
- 纯客户端架构 + 无用户体系 = 无法追踪 DAU/MAU/留存率。PRD 定义的 KR 和成功标准需要后端数据支撑，但架构决策移除了后端。这是一个严重的矛盾——我们无法衡量自己是否成功。
- 无社交分享（F010 Won't）、无邀请机制、无内容社区。产品的增长完全依赖自然流量和口碑，但没有设计任何口碑传播的触点。
- 无用户体系意味着无法做 A/B 测试、无法做用户分群、无法做个性化推荐。产品对所有用户展示完全相同的内容，这在"信息工具"类产品中是致命的——不同风险偏好的用户需要的选股策略完全不同。
- 数据不能跨设备同步，换手机等于重新开始。这对于需要持续跟踪的"关注列表"是严重缺陷。

**挑战**：
- 没有后端，你怎么知道日活 60% 还是 6%？
- 用户换手机后关注列表丢失，他会重新添加还是直接卸载？
- 所有用户看到一样的推荐列表，激进型和保守型投资者都满意吗？

**改进建议**：
- A) 引入最轻量的匿名事件上报（sentry_flutter 已在技术栈中，可扩展用于基础使用统计：打开次数、推荐/关注 Tab 切换次数、列表操作次数）（1 天）—— **推荐**
- B) 支持导出/导入关注列表（JSON 文件，通过文件分享），解决换手机数据迁移问题（0.5 天）
- C) 在推荐列表页底部增加"分享今日推荐"按钮，生成图片卡片分享到微信（v2 规划，当前不增加工作量，但需在 PRD 中记录）

**如果不改的代价**：
- 无法量化产品是否成功，2 周后无法做出基于数据的决策
- 换手机用户流失，无法积累长期用户

### 5. 设计质量 (8/10)

**现状**：
设计规范质量优秀。74/80 的审计得分（A 评级），AI Slop 检测 10 项全部通过。信息层级清晰（每屏一个主操作），数据格式统一，错误状态覆盖完整。组件规范详细到像素级别，包含 ASCII 线框图、状态变体描述、交互流程。

**亮点**：
- 涨跌色严格遵循 A 股标准（红涨绿跌），符合目标用户习惯
- 评分 Badge 的颜色分级（8-10 红 / 5-7 黄 / 1-4 绿）直觉且信息密度高
- 滑动操作（左滑删除、右滑置顶）符合移动端原生交互习惯
- 骨架屏、离线 Banner、空状态三件套全部覆盖，降级体验完整
- 免责声明规范清晰，合规意识到位

**问题**：
- 详情页信息密度极高（价格区 + 评分卡 + 公司信息 + 跟踪摘要 + 新闻列表 + 免责声明），在 375px 屏幕上需要大量滚动。对于"5 分钟快速浏览"的场景，详情页的效率可能不够。
- 推荐列表中"波段低谷"标签的位置设计规范说"紧跟在评分 Badge 右侧或股票名称下方"，两种位置表述模糊，需确认。
- 预警开关（Switch 组件）放在详情页右上角，但用户可能在列表页就想快速开关预警，需要回到详情页操作增加了 2 次点击。
- 响应式仅覆盖 375-428px，无平板适配。虽然 PRD 明确 Out of Scope，但如果未来需要 iPad 适配，当前的单列布局迁移成本较高。

**挑战**：
- 详情页滚动过长，核心信息（评分 + 预判）是否能在首屏内展示？
- 预警开关为什么不在列表页的滑动操作中？右滑只有"置顶"，没有"预警开关"？

**改进建议**：
- A) 详情页优化信息折叠：跟踪摘要默认折叠，点击展开；评分 + MA 值始终在首屏可见（0.5 天）—— **推荐**
- B) 列表页左滑操作增加第二个按钮："预警"（蓝色图标），与"删除"（红色）并列，减少进入详情页的频率（0.5 天）
- C) 统一"波段低谷"标签位置为"评分 Badge 右侧"（0 天，仅设计确认）

**如果不改的代价**：
- 详情页信息过载 = 用户找不到关键信息，增加认知负担
- 预警操作路径长 = 用户不愿意设置预警，F006 功能价值降低

## 关键风险警示

### 风险 1：东方财富 API 封禁（概率：中，影响：致命）
纯客户端架构的唯一数据源是非官方 API。如果东方财富加强反爬或变更接口格式，产品将完全不可用，且无法通过服务端代理快速切换（因为没有服务端）。
**缓解**：MVP 阶段就集成新浪财经作为实时备用数据源。

### 风险 2：核心假设未验证（概率：高，影响：高）
PRD 明确标注"假设 [未验证假设]"。所有用户画像、痛点描述、JTBD 场景都基于团队对散户投资者的推测，而非真实用户反馈。
**缓解**：上线前完成至少 3-5 个目标用户访谈。

### 风险 3：无法衡量成功（概率：高，影响：中）
纯客户端架构无法收集 DAU/MAU/留存等基础指标。PRD 定义的成功标准（日活 60%+ 查看推荐）无法验证。
**缓解**：通过 sentry_flutter 扩展最轻量的匿名事件上报。

## 总结

股势 TrendStock 的产品方向是合理的——为散户投资者提供一个聚焦波段分析的信息聚合工具。设计规范质量高，架构决策务实。但存在三个需要立即解决的关键问题：

1. **零用户验证**是最大的风险。在投入 2 周开发之前，必须花 0.5 天验证假设。
2. **东方财富 API 单点依赖**是最大的技术风险。必须集成备用数据源。
3. **无法衡量成功**是最大的商业风险。必须增加最轻量的数据收集。

平均分 6.8/10，需要完成上述调整后继续。不建议回滚——产品方向正确，执行层面需要加固。
